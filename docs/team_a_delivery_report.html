Testing Methodology Why & How We Test
Three tiers of testing ensure correctness at every level — from individual functions to full system behavior.

1 Unit Tests — Contract Validation

Verify that each module honors its API contract: correct inputs produce correct outputs, invalid inputs raise expected errors, and edge cases are handled gracefully.

  • 183 tests across 8 test files
  • Covers: config generation, directory creation, nginx config, NSSM services, FHIR adapter, EHR config, output sync, branding
  • Tool: pytest (Python 3.11+)
  • Why: Fast feedback loop — catches regressions in seconds, validates each component in isolation

2 Property-Based Tests — Formal Correctness

Instead of hand-picked examples, property-based tests generate random inputs and verify that invariants hold across the entire input space. This catches edge cases that example-based tests miss.

  • 54 tests validating 21 correctness properties
  • ~10,800 random examples generated per full run
  • Python: Hypothesis (200 examples per property)
  • TypeScript: fast-check (100 runs per property) + vitest
  • Why: Proves invariants like "FHIR round-trips are lossless", "conflict resolution never loses data", "NTFS ACLs grant exactly 2 principals" — things that a handful of examples can't guarantee

3 Integration / Checkpoint Tests

Checkpoint tasks (Tasks 4, 7, 10, 13) run the full test suite to confirm that all components work together after each milestone.

  • Phase 1 checkpoint: 79 tests pass
  • Phase 2 backend checkpoint: 170+ tests pass
  • Final checkpoint: 567 tests pass, 0 regressions
  • Why: Ensures that new code doesn't break existing functionality — catches integration issues between modules

🔬 Why Property-Based Testing?

Traditional example-based tests check a few known inputs. Property-based tests check thousands of randomly generated inputs against formal invariants.

  • Discovered a real bug: ConflictResolver.resolve used default cp1252 encoding on Windows — failed on non-ASCII clinical content (e.g., accented patient names). Fixed to explicit UTF-8.
  • Validates round-trip properties: serialize → deserialize → identical result
  • Validates monotonicity: incremental sync only fetches newer files
  • Validates safety: conflict resolution never silently drops data
  • Hypothesis automatically shrinks failing inputs to the minimal reproducing case
