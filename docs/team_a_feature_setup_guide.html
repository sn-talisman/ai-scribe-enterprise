Windows Deployment (PowerShell Script)

The script deploy/setup_provider_server.ps1 automates the full provider server setup on Windows.

What it does

✓ Verifies Python 3.11+, Node.js 18+, NSSM, nginx
✓ Creates venv with CPU-only deps (no GPU packages)
✓ Generates .env.provider + updates deployment.yaml
✓ Creates data directories with NTFS ACLs
✓ Configures nginx HTTPS reverse proxy (optional)
✓ Registers NSSM Windows Services (optional)

Basic usage

# Minimal — connects to pipeline at default localhost:8100
.\deploy\setup_provider_server.ps1

# Full production setup
.\deploy\setup_provider_server.ps1 `
  -PipelineUrl "http://192.168.1.50:8100" `
  -PublicUrl "https://provider.clinic.com" `
  -WithNginx `
  -TlsCert "C:\certs\provider.crt" `
  -TlsKey "C:\certs\provider.key" `
  -WithService

Parameters

Parameter Default Description
-PipelineUrl http://localhost:8100 Pipeline server URL
-PublicUrl (empty) This server's public-facing URL
-Secret (auto-generated) Inter-server auth secret. If omitted, a 64-char cryptographic secret is generated
-WithNginx off Install and configure nginx as HTTPS reverse proxy
-TlsCert (empty) Path to TLS certificate (used with -WithNginx)
-TlsKey (empty) Path to TLS private key (used with -WithNginx)
-WithService off Register NSSM Windows Services for API and web UI

Windows Services Created

Service Name Process Port Depends On
ai-scribe-api Uvicorn (FastAPI) 8000 —
ai-scribe-web Next.js 3000 ai-scribe-api
ai-scribe-nginx nginx 443 —

All services: automatic startup, 5-second restart delay on failure.

Managing Services

nssm start ai-scribe-api       # Start
nssm stop ai-scribe-api        # Stop
nssm status ai-scribe-api      # Check status
nssm get ai-scribe-api AppStdout  # View log path

EHR FHIR R4 Integration

The FHIR adapter (mcp_servers/ehr/fhir_server.py) connects to live EHR systems. Supports Epic, Cerner, and Athena.

Configuration

Edit config/deployment.yaml:

ehr:
  adapter: "fhir"          # Change from "stub" to "fhir"
  fhir:
    base_url: "https://fhir.epic.com/interconnect-fhir-oauth/api/FHIR/R4"
    vendor: "epic"          # "epic" | "cerner" | "athena"
    client_id: "your-client-id"
    client_secret_env: "AI_SCRIBE_EHR_CLIENT_SECRET"
    scopes:
      - "patient/*.read"
      - "patient/DocumentReference.write"

Set the client secret:

export AI_SCRIBE_EHR_CLIENT_SECRET=your-ehr-client-secret

Supported Operations

Operation FHIR Resource Method
Patient searchPatientGET (search by MRN)
Problem listConditionGET (active conditions)
MedicationsMedicationRequestGET (active meds)
AllergiesAllergyIntoleranceGET
Recent labsObservationGET (last 90 days)
Last visit noteDocumentReferenceGET (most recent)
Push approved noteDocumentReferencePOST (create)
⚠ Stub Fallback

If the FHIR endpoint is unreachable, the adapter automatically falls back to the stub EHR server. No config change needed.

🔒 OAuth2 Tokens

Tokens are cached and auto-refreshed 60s before expiry. On 401, the token is invalidated and retried once.

Keeping Stub Mode

To stay on local JSON/YAML files, leave the default:

ehr:
  adapter: "stub"

Output Sync Enhancements

Three improvements to how generated notes flow from the pipeline server to the provider server.

⚡

WebSocket Sync

Instant notification when pipeline completes. Falls back to polling on disconnect.

📥

Incremental Sync

Only downloads files with newer remote timestamps. Uses If-Modified-Since headers.

🔄

Conflict Resolution

When both versions are modified: keep both, keep remote, or keep local.

Configuration

In config/deployment.yaml:

output_sync_enhanced:
  websocket_enabled: true            # Enable WebSocket on-demand sync
  websocket_reconnect_interval: 30   # Seconds between reconnect attempts
  incremental: true                  # Only fetch newer files
  conflict_strategy: "keep_both"     # "keep_both" | "keep_remote" | "keep_local"
Conflict strategies: keep_both renames local with .local suffix and writes remote at original path. keep_remote overwrites local. keep_local ignores remote.

Correction Diff Engine

When a provider edits a note, a structured diff is computed and saved as training data for the learning loop.

What it captures

📌 Change type (add / delete / modify)
📋 SOAP section identification
📝 Original & edited text
🔢 Line numbers
🔒 SHA-256 hashes of both versions
🕓 Timestamp

Output Format

Saved as correction_diff.json in the encounter's output directory:

{
  "encounter_id": "sample_001",
  "original_hash": "a1b2c3...",
  "edited_hash": "d4e5f6...",
  "timestamp": "2026-03-22T10:30:00.000Z",
  "changes": [
    {
      "type": "modification",
      "section": "Assessment",
      "original_text": "Acute cervical strain",
      "edited_text": "Acute cervical strain, improving",
      "line_start": 42,
      "line_end": 42
    }
  ]
}

This data feeds the learning loop for per-provider ASR and note generation improvement.

Practice Branding

Customize the provider portal with practice-specific branding.

Configuration

Edit config/deployment.yaml:

branding:
  practice_name: "Excelsia Injury Care"
  logo_url: "/images/practice-logo.png"
  primary_color: "#2c3e50"

What it affects

🏧
Practice name in sidebar header
🎨
Logo in navigation bar
🎨
Primary color on UI accents

API Endpoint

GET /config/branding returns:

{ "practice_name": "Excelsia Injury Care", "logo_url": "/images/practice-logo.png", "primary_color": "#2c3e50" }
If no branding is configured, the portal shows "AI Scribe" with the default blue theme (#1a5276).

Configuration Reference

All Team A features are configured in config/deployment.yaml. Pydantic models live in config/deployment.py.

Environment Variables

Variable Required Description
AI_SCRIBE_SERVER_ROLE Yes provider-facing, processing-pipeline, or both
AI_SCRIBE_INTER_SERVER_SECRET Production Shared secret for inter-server authentication
AI_SCRIBE_EHR_CLIENT_SECRET If using FHIR OAuth2 client secret for EHR integration

AI_SCRIBE_DATA_DIR No Override data directory path
AI_SCRIBE_OUTPUT_DIR No Override output directory path
AI_SCRIBE_CONFIG_DIR No Override config directory path
